# Learning Journal - Developing My QA Strategy

**Date:** July 11, 2026

## Reflection

Today I shifted my thinking from "learning QA" to "thinking like a QA engineer."

Initially, I wanted to continue creating more documentation, but I realized that documentation only has value when it supports actual testing. The purpose of my QA Toolkit is not to become a large collection of templates—it is to provide a structured process for testing real software projects.

## My QA Workflow

I established the following workflow for future projects:

1. Build the software.
2. Create a project-specific QA folder.
3. Write a test plan.
4. Execute smoke testing.
5. Execute regression testing after changes.
6. Record defects using standardized bug reports.
7. Verify fixes.
8. Document testing results.

This workflow mirrors how professional QA teams approach software releases and gives me a repeatable process that I can improve over time.

## Personal Development

I also recognized that my own projects are ideal practice environments. Since I understand the intended functionality, I can focus on developing strong testing habits before moving on to more complex applications.

As my skills improve, I plan to expand into accessibility testing, API testing, database validation, automated testing, and continuous integration.

## Looking Ahead

My immediate priority is completing my personal website. Once it is functional, I will use it as my first formal QA case study. After that, I will apply the same process to the Neural Mechanika project, allowing me to compare my growth as both a developer and a QA engineer.
